The problem

Applying faster doesn't fix weak positioning.

Most career tools help you send more applications. The real bottleneck is usually the candidate profile, not the application volume.

📄

Generic resumes don't prove role fit

A resume that lists years of experience does not show a hiring manager that you have built AI copilot UX, RAG citation flows, or enterprise approval workflows.

🐙

GitHub often hides your target role

A profile full of side projects and old repos rarely signals the specific occupation clearly enough for a recruiter scanning for 30 seconds.

📁

Portfolio projects don't map to proof

Building something is not the same as building something that proves a specific skill to a specific interviewer. Most portfolio projects are not scoped to answer interview questions.

📆

Career plans aren't executable weekly

Strategy documents that live in Notion and never become a Monday-to-Friday task list do not produce hireable candidates. Execution is the missing layer.

🎯

Positioning is unclear without a gap map

Without knowing exactly which proof signals are missing, effort goes to the wrong places. A role gap analysis turns vague ambition into a prioritized fix list.

What it does

Six workflows. One local system.

Each command generates a standalone Markdown file in output/. No API key, no account, no internet required.

Profile Audit

Review how your current GitHub, resume, and portfolio appear to a recruiter scanning for your target occupation.

  • Strengths to surface immediately

  • Next five actions to take

How it works

Profile to proof in one local workflow.

Edit profile.yml with your real details. Each command reads it and writes a fresh Markdown report to output/.

profile.yml npm run init
→
Profile Audit npm run audit:profile
→
Role Gaps npm run gap:role
→
Portfolio Plan npm run portfolio:plan
→
GitHub Growth npm run github:growth
→
Interview Stories npm run interview:stories
→
Weekly Plan npm run plan:weekly

All outputs are local Markdown files. Edit, commit, or delete them. Run any command again to regenerate from scratch.

# Truthfulness Guardrails
Applied to every generated report.

Do not claim production usage unless the project is deployed and in use.
Do not claim merged open-source PRs unless the PR was merged.
Label mock services as mock services in every README.
Label planned features as planned in roadmaps, not in proof summaries.
Do not invent metrics, user counts, stars, or testimonials.
Only publish claims you can defend clearly in a technical interview.

## Claims To Avoid Unless Proven
"We scaled to X users" — only if you can name the metric source.
"I led the architecture" — only if you made the key decisions.
"The project is in production" — only if it is actually deployed.
"I contributed to open source" — only if the PR was merged.

FAQ

Common questions.

Is this an auto-apply bot?

No. Occupation-Ops does not browse job boards, access employer websites, or submit applications. It generates local Markdown files you review and act on yourself. The npm run doctor output explicitly confirms: Automatic applications: no.

Does it require API keys?

No. All commands run with Node.js on your local machine. No API key, no account, no internet connection is required. The v0.3 roadmap includes an optional AI provider integration, but local-first will always be the default.

Can non-developers use it?

The CLI commands require Node.js and a terminal, so some developer familiarity helps. However, the Markdown outputs — role gap analysis, portfolio plan, interview stories — are readable and useful to anyone, including career coaches, mentors, and job seekers reviewing their own positioning.

Can I add my own occupation track?

Yes. Occupation tracks are Markdown files in the tracks/ directory. Copy any existing track, follow the structure (proof signals, gap audit, portfolio ideas, interview topics), and submit a pull request. Good-first-issue guidance is in CONTRIBUTING.md.

Is it honest and safe to use?

Yes. Every report includes a Truthfulness Guardrails section that explicitly tells you what not to claim. The tool is designed to help you build real, verifiable proof — not fabricate credentials. It will not invent metrics, testimonials, or work history.

How is this different from Career-Ops?

Occupation-Ops is an independent project with different positioning, commands, output structure, and design. It focuses on becoming hireable through local proof-building workflows rather than any specific author's personal story. It does not share code, content, screenshots, metrics, or community links with Career-Ops.
