# Workflow Contract

Occupation-Ops only works well when the inputs and outputs remain truthful, local, and reviewable.

## Required input contract

Users should provide:

- target occupation
- realistic target role titles
- real links
- current proof projects
- skills already demonstrated
- skills still missing
- weekly time constraints

## Output contract

Outputs should help the user create:

- clearer positioning
- better GitHub proof
- stronger project planning
- better interview stories
- more consistent weekly execution

Outputs must not:

- invent experience
- imply hiring outcomes
- claim ATS optimization guarantees
- automate mass outreach
- hide what is mocked, planned, or incomplete

## Review contract

Before publishing any generated output, the user should ask:

1. Can I defend this claim in an interview?
2. Is this based on something I actually built, shipped, tested, or learned?
3. Did I mark planned work as planned?
4. Did I mark mock systems as mock systems?

## Contributor contract

Contributors should preserve:

- local-first behavior
- ethical career positioning
- visible truthfulness guardrails
- practical, developer-oriented outputs
