# Occupation Contract

Occupation-Ops uses an occupation contract to keep career planning specific,
truthful, and execution-oriented.

## Required Inputs

- Target occupation
- Target seniority
- Target geography or remote preference
- Current resume summary
- Current GitHub or portfolio links
- Existing proof projects
- Skills already demonstrated
- Skills not yet demonstrated
- Constraints, such as time per week and preferred stack

## Required Outputs

- Role positioning statement
- Current proof inventory
- Missing proof signals
- Portfolio project plan
- GitHub and LinkedIn positioning notes
- Interview topic map
- 30-day roadmap
- Weekly execution checklist

## Truthfulness Rules

- Say "planned" for planned work.
- Say "prototype" for prototypes.
- Say "mock" for mock integrations.
- Say "research notes" for research notes.
- Do not imply production usage unless proven.
- Do not imply merged PRs unless actually merged.

## Review Questions

- Would a recruiter understand the target role in 30 seconds?
- Would a hiring manager see role-specific proof?
- Would a maintainer know what contribution help is being offered?
- Would the user be comfortable defending every claim in an interview?
